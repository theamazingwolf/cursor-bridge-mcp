# Phase 2: Censi Bridge Sync

> **Cursor Bridge MCP** — PRD Phase 2 of 2

---

## Phase Overview

### Goals

This phase adds the Censi-to-Cursor bridge — an MCP tool that reads context documents generated by Censi (the companion context engine) and converts them to `.mdc` files in `.cursor/rules/` for native Cursor consumption. This enables a bidirectional workflow where Censi generates context from code analysis and Cursor Bridge makes that context available in Cursor's native format. Only documents with `generated: true` in their frontmatter are synced, preserving hand-authored Cursor rules.

### Scope

| In scope | Out of scope |
|----------|--------------|
| `sync_from_censi` MCP tool | Automatic/scheduled sync (manual trigger only) |
| Reading Censi output directory for generated context docs | Censi installation or configuration |
| Converting Censi frontmatter to Cursor `.mdc` frontmatter | Syncing non-generated (hand-authored) Censi docs |
| Writing `.mdc` files to `.cursor/rules/` | Reverse sync (Cursor rules -> Censi format) |
| Dry-run mode for previewing sync operations | Syncing agents or skills from Censi (rules only) |
| Change detection (skip unchanged files) | Watching Censi output directory for auto-sync |
| Censi output directory auto-detection via `.censi/config.yaml` | |
| Configuration via `.cursor-bridge/config.yaml` censi section | |

### Dependencies

- **Phase 1** — Core server, context index, configuration loading, and MCP tool registration framework.

## User Stories

### US-2.1 — Manual Censi-to-Cursor Sync

**As a** developer using both Censi and Cursor,
**I want** to sync Censi's generated context documents into `.cursor/rules/` as `.mdc` files,
**so that** Cursor's native AI can read context produced by Censi's code analysis.

#### Acceptance Criteria

| # | Given / When / Then |
|---|---------------------|
| AC-1 | Given a Censi output directory containing `referral-system.md` with `generated: true` in frontmatter, when `sync_from_censi` is called with that directory, then `.cursor/rules/referral-system.mdc` is created with Cursor-compatible frontmatter. |
| AC-2 | Given a Censi doc with `description`, `globs`, and `alwaysApply` in its frontmatter, when synced, then the `.mdc` file preserves all three fields in its YAML frontmatter. |
| AC-3 | Given a Censi doc without `generated: true` in frontmatter, when `sync_from_censi` is called, then the doc is skipped and not synced. |
| AC-4 | Given a previously synced `.mdc` file whose source Censi doc has not changed (same content hash), when `sync_from_censi` is called, then the file is skipped with reason `"unchanged"`. |
| AC-5 | Given a previously synced `.mdc` file whose source Censi doc has been updated, when `sync_from_censi` is called, then the `.mdc` file is overwritten with the updated content. |

## Functional Data Requirements

No new persistent data structures. This phase reads existing Censi markdown files and writes `.mdc` files. The sync state (which files have been synced) is determined at runtime by comparing source file content hashes against existing `.mdc` files — no separate state file is maintained.

### Censi Document Format (Input)

Censi output files are markdown with YAML frontmatter:

```markdown
---
description: "Human-readable description"
globs: "src/referrals/**"
alwaysApply: false
generated: true
censi_version: "1.0"
source_hash: "abc123"
---

# Document Title

Markdown content...
```

The `generated: true` field is the gate — only documents with this flag are eligible for sync.

### MDC File Format (Output)

Standard Cursor `.mdc` format:

```markdown
---
description: "Human-readable description"
globs: "src/referrals/**"
alwaysApply: false
---

# Document Title

Markdown content...
```

The Censi-specific fields (`generated`, `censi_version`, `source_hash`) are stripped during conversion. The markdown body is preserved as-is.

## API / Endpoint Specs

### `sync_from_censi`

Reads context docs from Censi's output directory and converts them to `.mdc` files in `.cursor/rules/`.

**Input schema**:

```json
{
  "censi_output_dir": {
    "type": "string",
    "description": "Path to Censi output directory relative to workspace root. Auto-detected from .censi/config.yaml if omitted."
  },
  "target_dir": {
    "type": "string",
    "default": ".cursor/rules",
    "description": "Target directory for .mdc files relative to workspace root"
  },
  "dry_run": {
    "type": "boolean",
    "default": false,
    "description": "If true, report planned actions without writing files"
  }
}
```

**Success response**:

```json
{
  "synced": [
    {
      "source": "context/referral-system.md",
      "target": ".cursor/rules/referral-system.mdc",
      "action": "updated"
    },
    {
      "source": "context/api-conventions.md",
      "target": ".cursor/rules/api-conventions.mdc",
      "action": "created"
    }
  ],
  "skipped": [
    {
      "source": "context/main-overview.md",
      "reason": "unchanged"
    },
    {
      "source": "context/notes.md",
      "reason": "not generated (missing generated: true)"
    }
  ],
  "summary": {
    "created": 1,
    "updated": 1,
    "unchanged": 1,
    "skipped_not_generated": 1,
    "dry_run": false
  }
}
```

**Error responses**:

| Condition | Error message |
|-----------|--------------|
| Censi output directory does not exist | `"Censi output directory 'context' not found at {absolute_path}"` |
| Target directory does not exist and cannot be created | `"Cannot create target directory '.cursor/rules': {error}"` |
| No `.md` files found in Censi output directory | `"No markdown files found in Censi output directory 'context'"` |

**Internal flow**:

```
1. Resolve censi_output_dir:
   a. If provided, use it
   b. Else if .censi/config.yaml exists, read output_directory from it
   c. Else default to "context"
2. Validate censi_output_dir exists
3. Ensure target_dir exists (create if needed and not dry_run)
4. List all .md files in censi_output_dir
5. For each file:
   a. Parse YAML frontmatter
   b. If generated !== true, add to skipped (reason: not generated), continue
   c. Compute content hash (SHA-256 of full file content)
   d. Determine target path: target_dir/{filename}.mdc
   e. If target .mdc exists:
      - Read existing .mdc content
      - Compare markdown body (ignoring frontmatter differences)
      - If unchanged, add to skipped (reason: unchanged), continue
      - Else mark as "updated"
   f. Else mark as "created"
   g. If not dry_run:
      - Build .mdc frontmatter (strip generated, censi_version, source_hash)
      - Write .mdc file
   h. Add to synced list with action
6. Rebuild context index to pick up new/changed rules
7. Return synced, skipped, and summary
```

## Technical Considerations

### Frontmatter Field Stripping

When converting Censi docs to `.mdc`, strip Censi-specific fields (`generated`, `censi_version`, `source_hash`) from the frontmatter. Preserve all Cursor-compatible fields (`description`, `globs`, `alwaysApply`). Use the same `yaml` package from Phase 1 to parse and re-serialize the frontmatter, ensuring valid YAML output.

### Content Comparison for Change Detection

Compare the markdown body (everything below the frontmatter) between source and target to determine if a sync is needed. Do not compare frontmatter, since Censi-specific fields are stripped during conversion — comparing full content would always show a difference. Use a simple string comparison after normalizing line endings to `\n`.

### Target Directory Auto-Creation

If the target directory (`.cursor/rules/`) does not exist when `sync_from_censi` is called (not in dry-run mode), create it. This handles the case where a project uses Censi but hasn't set up Cursor rules yet.

### Security

- **Write scope**: The tool only writes to the configured `target_dir` (default `.cursor/rules/`). Path traversal in filenames is prevented by using only the basename of source files.
- **No execution**: Synced content is written as static `.mdc` files — never executed.
- **Censi trust**: The tool trusts Censi output files as authored content. It does not validate or sanitize markdown content beyond frontmatter processing.

## Risks and Open Questions

| # | Item | Impact | Mitigation |
|---|------|--------|------------|
| R-1 | Sync overwrites hand-edited `.mdc` files if they share a name with a Censi-generated doc | User loses manual edits to Cursor rules | Mitigate: only sync files where the source has `generated: true`. Document that hand-authored rules should use names that don't conflict with Censi output filenames. |
| R-2 | Censi config format (`output_directory` field name) may change across Censi versions | Auto-detection breaks silently | Mitigate: fall back to `"context"` default if `.censi/config.yaml` exists but lacks expected fields. Log a warning. |
| Q-1 | Should sync preserve a marker in the `.mdc` frontmatter indicating it was generated by Censi Bridge? | Would help identify which `.cursor/rules/` files are safe to overwrite on next sync | Decided: yes, add a `# Generated by cursor-bridge — do not edit manually` comment at the top of synced `.mdc` files. This is a comment, not a frontmatter field, so Cursor ignores it. |
| Q-2 | Should the tool support syncing entire subdirectory trees from Censi, or only flat file lists? | Censi may organize output into subdirectories in the future | Interim: flat files only. Revisit if Censi adds subdirectory output. |
